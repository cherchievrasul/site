# Pitfalls Research

**Domain:** Personal static website on shared hosting (Beget)
**Researched:** 2026-06-24
**Confidence:** HIGH

## Critical Pitfalls

### Pitfall 1: FTP deployment overwrites or misplaces files

**What goes wrong:**
Файлы загружаются не в ту директорию, либо при повторной загрузке теряются изменения. На Beget корневая директория сайта — `public_html` (или `ваш_домен/public_html`). Если положить `index.html` в корень FTP-аккаунта (рядом с `public_html`), сайт не откроется — браузер покажет пустую страницу, ошибку 403 или дефолтную заглушку хостинга.

Другая частая ошибка — FTP-клиент (FileZilla, Total Commander) по умолчанию загружает файлы в бинарном режиме, но текстовые файлы (HTML, CSS, JS) должны передаваться в ASCII-режиме. При неверном режиме могут поплыть переводы строк и кодировка.

**Why it happens:**
Структура папок Beget неочевидна для новичков. FTP-доступ даёт доступ к домашней директории аккаунта, где `public_html/` — лишь одна из многих папок (рядом лежат `cgi-bin`, `logs`, `tmp`). Пользователь не сверяется с документацией хостинга о том, куда именно класть файлы.

**How to avoid:**
1. Всегда загружать файлы сайта в `public_html/` (или `домен/public_html/` если на аккаунте несколько доменов)
2. В FileZilla: `Transfer → Transfer Type → Auto` (автоопределение ASCII/бинарного)
3. После загрузки сразу открыть домен в браузере и убедиться, что видна твоя страница, а не заглушка хостинга
4. Создать в проекте файл `deploy.md` с точным путём для FTP-загрузки

**Warning signs:**
- При открытии домена видна страница «Сайт успешно создан» или «Здесь будет сайт» от Beget
- Вместо сайта — ошибка 403 Forbidden
- `index.html` лежит на сервере, но сайт не открывается

**Phase to address:**
Phase 1 (Landing Page) — первый же деплой должен быть в правильную директорию

### Pitfall 2: Кодировка кириллицы — кракозябры вместо русского текста

**What goes wrong:**
Русский текст на странице отображается как `?????? ???????` или `����� ��������`. Самая частая жалоба на форумах Beget. Проблема возникает на трёх уровнях: (1) сам HTML-файл сохранён в неправильной кодировке, (2) сервер Beget отдаёт неверный HTTP-заголовок `Content-Type`, (3) `<meta charset>` в HTML конфликтует с серверным заголовком.

**Why it happens:**
Beget по умолчанию отправляет HTTP-заголовок `Content-Type: text/html; charset=windows-1251` (историческая настройка для российского хостинга). Если HTML-файл сохранён в UTF-8 (что стандарт для современных редакторов — VS Code, Sublime, Notepad++), браузер получает противоречивые указания: сервер говорит windows-1251, HTML говорит UTF-8. Результат — кракозябры.

Плюс Windows-версия Notepad сохраняет файлы в UTF-8 с BOM (Byte Order Mark) — три невидимых байта в начале файла, которые ломают отображение в браузере.

**How to avoid:**
1. **Всегда сохранять HTML-файлы в UTF-8 без BOM** (в VS Code: внизу справа клик на кодировку → «Save with Encoding» → «UTF-8»)
2. Добавить в `<head>` метатег `<meta charset="UTF-8">` **самым первым** (до `<title>`)
3. Создать файл `.htaccess` в `public_html/` с содержимым:
   ```apache
   AddDefaultCharset UTF-8
   ```
   Это переопределит серверную кодировку Beget на UTF-8
4. Проверить в браузере: F12 → Network → клик на HTML-файл → вкладка Headers → `Content-Type` должен содержать `charset=UTF-8`

**Warning signs:**
- Русские буквы отображаются как знаки вопроса или ромбики
- В разных браузерах текст выглядит по-разному
- В исходном коде страницы (Ctrl+U) русский текст нормальный, а в отображении — нет

**Phase to address:**
Phase 1 (Landing Page) — с самого первого HTML-файла

### Pitfall 3: Затянувшаяся «заглушка» — сайт не обновляется после повторной загрузки

**What goes wrong:**
Залил новую версию `index.html` через FTP, открыл сайт — а там старая версия. Очистил кэш браузера — не помогло. Думаешь, что FTP не работает, начинаешь перезаливать файлы, менять настройки — а проблема в другом.

**Why it happens:**
Три возможные причины:
1. **FTP-клиент не перезаписал файл.** Некоторые клиенты (особенно Total Commander) по умолчанию пропускают файлы с одинаковым размером или датой. Ты думаешь, что залил — а на сервере старая версия.
2. **Кэширование на стороне Beget.** Beget использует nginx в качестве фронтенда перед Apache. Статические файлы могут кэшироваться на уровне nginx на несколько минут. Плюс в панели управления Beget есть отдельная настройка кэширования.
3. **DNS-кэш интернет-провайдера.** После смены NS-записей или первого деплоя обновление DNS может идти до 48 часов (на практике — 2-8 часов для RU-зоны).

**How to avoid:**
1. В FTP-клиенте всегда выбирать «Overwrite» / «Перезаписывать» при конфликте
2. После загрузки проверить файл через файловый менеджер в панели управления Beget — так ты видишь реальное содержимое на сервере
3. Добавить в `.htaccess` отключение кэша для разработки:
   ```apache
   <FilesMatch "\.(html|css|js)$">
       Header set Cache-Control "no-cache, must-revalidate"
   </FilesMatch>
   ```
   Убрать, когда сайт готов!
4. Для проверки использовать приватный режим браузера (Ctrl+Shift+N)
5. При смене CSS/JS — добавить версионирование: `style.css?v=2`

**Warning signs:**
- Сайт не меняется после FTP-загрузки, даже через минуту
- В файловом менеджере Beget файл новый, в браузере — старый
- На другом устройстве / в приватном режиме сайт обновлён

**Phase to address:**
Phase 1 (Landing Page) — при первых деплоях

### Pitfall 4: Отсутствие .htaccess → ошибки 404, 403 и отсутствие ЧПУ

**What goes wrong:**
Сайт работает, но при попытке открыть `/about.html` — ошибка 404, хотя файл лежит на сервере. Или открывается `/about/` а там 403. Или `https://` версия сайта не открывается, хотя `http://` работает. Всё это решается через `.htaccess`, который новички часто не создают.

**Why it happens:**
На Beget Apache обрабатывает запросы, и `.htaccess` — это способ управлять поведением сервера на уровне директории. По умолчанию Beget имеет свои настройки (например, `DirectoryIndex index.html index.php`), но без `.htaccess` нет:
- ЧПУ (человекопонятных URL) — только `.html` на конце
- Редиректа с HTTP на HTTPS
- Кастомной страницы 404
- Настройки кодировки
- Кэширования
- Защиты от прямого листинга директорий

**How to avoid:**
1. Создать `.htaccess` **сразу** при первом деплое, а не «потом добавлю»
2. Минимальный `.htaccess` для статического сайта:
   ```apache
   # Кодировка
   AddDefaultCharset UTF-8

   # Индексный файл
   DirectoryIndex index.html

   # Редирект HTTP → HTTPS (после установки SSL)
   RewriteEngine On
   RewriteCond %{HTTPS} off
   RewriteRule ^(.*)$ https://%{HTTP_HOST}/$1 [R=301,L]

   # Убираем .html в URL (ЧПУ)
   RewriteCond %{REQUEST_FILENAME} !-d
   RewriteCond %{REQUEST_FILENAME}\.html -f
   RewriteRule ^(.*)$ $1.html [L]

   # Кастомная страница 404
   ErrorDocument 404 /404.html
   ```
3. В Windows `.htaccess` не создаётся через «Создать файл» (Windows считает, что файл без имени — это ошибка). Создавать через VS Code: File → New File → Save As → ввести `.htaccess`
4. FTP-клиент должен показывать скрытые файлы (в FileZilla: Server → Force showing hidden files)

**Warning signs:**
- URL без `.html` на конце не открывается
- `https://` версия не работает
- При запросе `/non-existent` — стандартная ошибка Apache вместо красивой страницы
- Содержимое директории видно при заходе на `/images/`

**Phase to address:**
Phase 1 (Landing Page) — `.htaccess` закладывается с первого деплоя

### Pitfall 5: Тяжёлые изображения — сайт грузится 10+ секунд

**What goes wrong:**
Статический сайт с 3-5 «красивыми» картинками. Фотографии с телефона весят по 5-8 МБ каждая. Никто не сжимает перед загрузкой — «сайт же статический, он и так быстрый». Результат: страница грузится 8-15 секунд на мобильном интернете, посетители уходят не дождавшись.

**Why it happens:**
Новички думают, что «статический = быстрый по умолчанию». На самом деле статический сайт без оптимизации изображений может быть медленнее, чем WordPress с плагином кэширования. Современные телефоны снимают в разрешении 4000×3000 при 300 DPI — такие изображения на веб-странице никогда не нужны в исходном размере.

**How to avoid:**
1. **Ресайзить изображения до реального размера отображения** (максимум 1920px по ширине для full-width, 800px для контентных)
2. **Сжимать JPEG до качества 70-80%** (визуально незаметно, размер падает в 3-5 раз)
3. **Конвертировать в WebP** (поддержка 97% браузеров в 2026, размер на 30-40% меньше JPEG)
4. Инструменты: `squoosh.app` (онлайн), `imagemagick` (консоль), `sharp` (Node.js)
5. Правило: ни одно изображение на странице-заглушке не должно весить больше 200 КБ
6. Для иконок и логотипов — SVG (вектор, идеальное качество на любом экране, минимальный вес)

**Warning signs:**
- Страница грузится больше 3 секунд на 3G
- Lighthouse (F12 → Lighthouse) показывает красный для Performance
- Размер страницы > 2 МБ

**Phase to address:**
Phase 1 (Landing Page) — с первого деплоя

### Pitfall 6: Файлы остаются после переименования — мусор на хостинге

**What goes wrong:**
Переименовал `index-old.html` в `index.html` локально. Загрузил на сервер. На сервере теперь лежат ОБА файла — `index-old.html` и `index.html`. Через месяц на хостинге десятки «старых» файлов, непонятно какие актуальны. При этом старые файлы могут быть доступны по прямым ссылкам.

**Why it happens:**
FTP-клиенты по умолчанию НЕ удаляют файлы на сервере, которых нет в локальной папке. Это не синхронизация, а односторонняя загрузка. Пользователь думает, что «перезаписал», а на самом деле — добавил новый файл рядом со старым.

**How to avoid:**
1. **Использовать синхронизацию вместо копирования.** В FileZilla и WinSCP есть режим «Synchronised browsing» + «Sync folders». В Total Commander: «Commands → Synchronize Dirs»
2. Перед синхронизацией ВСЕГДА сравнивать папки — на сервере могут быть файлы, которых нет локально (`.htaccess`, логи)
3. Вести список файлов сайта в проекте — сразу видно, что должно быть на сервере
4. Альтернатива: удалять ВСЁ содержимое `public_html/` (кроме `.htaccess`) и заливать заново — для статического сайта это безопасно и быстро

**Warning signs:**
- В файловом менеджере Beget больше файлов, чем в локальном проекте
- Старые страницы открываются по прямым ссылкам, хотя «удалены»
- Закончилось дисковое пространство на хостинге

**Phase to address:**
Phase 1 (Landing Page) — процесс деплоя

### Pitfall 7: SSL не работает или Mixed Content после подключения HTTPS

**What goes wrong:**
Beget предоставляет бесплатный SSL от Let's Encrypt. Подключил — сайт открывается по HTTPS, но браузер показывает «Небезопасно» (жёлтый замочек вместо зелёного). На странице явно прописаны ссылки на ресурсы через `http://` (изображения, шрифты, внешние скрипты), и браузер блокирует «смешанный контент».

**Why it happens:**
При разработке локально протокол не важен — `http://` работает. Но после подключения SSL все ресурсы должны загружаться через HTTPS. Жёстко закодированные `http://` в HTML/CSS ломают безопасное соединение. Плюс Beget требует ручного продления Let's Encrypt каждые 90 дней, если не настроен авторедирект.

**How to avoid:**
1. Все внешние ресурсы (Google Fonts, CDN, изображения) подключать через `https://` с самого начала
2. Для локальных ресурсов использовать относительные пути (`/images/photo.jpg`) или протокол-независимые (`//example.com/script.js`)
3. После подключения SSL на Beget проверить: F12 → Console — искать ошибки Mixed Content
4. Обязательно добавить в `.htaccess` редирект HTTP → HTTPS (см. Pitfall 4)
5. Настроить автообновление SSL в панели Beget (если поддерживается тарифом)

**Warning signs:**
- Жёлтый треугольник вместо зелёного замочка
- Картинки не грузятся на HTTPS-версии
- В консоли браузера: `Mixed Content: The page at '...' was loaded over HTTPS, but requested an insecure...`

**Phase to address:**
Phase 2 (UI/Design) или при первом подключении SSL

### Pitfall 8: Мобильная версия не тестируется на реальных устройствах

**What goes wrong:**
Страница выглядит идеально в Chrome на десктопе при ширине 1920px. Открываешь на реальном телефоне — текст налезает на край, кнопки не нажимаются, шрифт микроскопический. Потому что тестировалось только через Chrome DevTools в режиме эмуляции (который не отражает реального рендеринга на iOS Safari и старых Android).

**Why it happens:**
DevTools эмулирует viewport, но не движок рендеринга. iOS Safari и Android WebView имеют свои особенности: обработка `vh` единиц, `position: fixed`, `backdrop-filter`, поддержка шрифтов. Плюс на реальных устройствах другой DPR (device pixel ratio), из-за чего границы в 1px могут выглядеть как 2px.

**How to avoid:**
1. Обязательный `<meta name="viewport" content="width=device-width, initial-scale=1.0">` в `<head>`
2. Использовать реальный телефон для финальной проверки (хотя бы один Android и один iOS)
3. Если нет второго устройства — открыть сайт на своём телефоне в разных браузерах (Chrome, Safari, Samsung Internet)
4. При локальной разработке использовать `npx serve` и заходить на локалку с телефона по IP-адресу в локальной сети
5. Минимальные брейкпоинты: 320px (маленький телефон), 768px (планшет), 1024px+ (десктоп)

**Warning signs:**
- Текст выходит за границы экрана на телефоне
- Кнопки слишком маленькие для пальца (< 44×44 px — рекомендация Apple HIG)
- Горизонтальная прокрутка на мобильном
- Шрифт меньше 16px на телефоне (iOS Safari зумит поля ввода с мелким шрифтом)

**Phase to address:**
Phase 2 (UI/Design) — при добавлении CSS стилей

## Technical Debt Patterns

Shortcuts that seem reasonable but create long-term problems.

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Не создавать `.htaccess` — «сайт простой, само заработает» | Экономия 5 минут | Отсутствие редиректов, кодировка ломается, нет ЧПУ. При добавлении страниц — тупик | **Никогда.** Даже для одной страницы `.htaccess` с кодировкой обязателен |
| Картинки в исходном разрешении «как с телефона» | Экономия времени на сжатии | Страница грузится 10+ секунд, посетители уходят | **Никогда.** Сжатие изображений — часть процесса вёрстки |
| `<br><br>` вместо параграфов и отступов | Быстро «накидать текст» | Невозможно стилизовать CSS. При добавлении контента всё разъезжается | **Никогда.** С первого дня — семантическая вёрстка |
| Inline-стили в атрибутах `style="..."` | Быстро «подкрасить» элемент | Каскад ломается, переопределить без `!important` нельзя. При доработке UI — кошмар | Только в Phase 1 (заглушка) как временное решение. В Phase 2 переписать на CSS |
| `alert()` для отладки вместо console.log | Моментально видно результат | Забытый `alert()` в продакшене блокирует загрузку страницы и выглядит ужасно | **Никогда.** `console.log` + F12 |
| Абсолютные пути в ссылках: `/C:/Users/.../image.jpg` | «Работает на моём компьютере» | На сервере все ссылки битые | **Никогда.** Только относительные пути |

## Integration Gotchas

Common mistakes when connecting to external services.

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| FTP (FileZilla / Total Commander) | Заливка в корень FTP, а не в `public_html/` | Проверить в панели Beget точный путь к директории сайта. Обычно `/home/логин/public_html/` |
| FTP — перезапись файлов | Клиент настроен на «Skip if same size», старые файлы не обновляются | В настройках клиента: Transfer → File exists → Overwrite |
| DNS Beget | После привязки домена сайт не открывается — ждать не надо, проблема в другом | Проверить через `ping домен` — если пингуется IP Beget, проблема не в DNS. Если нет — проверить NS-записи в панели Beget и у регистратора домена |
| SSL Let's Encrypt на Beget | Сертификат выпущен, но сайт не открывается по HTTPS | Проверить, что в `.htaccess` нет редиректа HTTP→HTTPS ДО установки сертификата. Порядок: сначала сертификат → потом редирект |
| Яндекс.Метрика / Google Analytics | Код вставлен, но статистика не собирается | Проверить, что скрипт загружается без ошибок (F12 → Console → нет блокировок). Возможно, блокировщик рекламы вырезает трекер — проверять с отключённым AdBlock |
| Файловый менеджер Beget | Редактирование файлов прямо в браузере ломает кодировку | Редактировать ТОЛЬКО локально в редакторе кода, затем заливать через FTP |

## Performance Traps

Patterns that work at small scale but fail as usage grows.

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Несжатые изображения | Lighthouse Performance < 50, LCP > 4s | См. Pitfall 5 — сжатие + WebP | Сразу на первом посетителе с мобильным интернетом |
| Шрифты с Google Fonts без `font-display: swap` | Текст не виден 3-5 секунд пока грузится шрифт | Добавить `&display=swap` в URL шрифта: `https://fonts.googleapis.com/css2?family=Roboto&display=swap` | При медленном соединении или если Google Fonts заблокирован |
| Много отдельных CSS-файлов | Много HTTP-запросов, долгая загрузка | Объединить в один `style.css`. Для production — минифицировать | 5+ CSS-файлов на странице |
| Отсутствие `loading="lazy"` на изображениях | Все картинки грузятся сразу, даже те, что ниже экрана | Добавить `loading="lazy"` к `<img>` под сгибом | Страницы с 5+ изображениями |
| PHP-скрипт для статики (ради формы обратной связи) | Страница перестала быть статической — уязвимости, медленная загрузка | Форма через сторонний сервис (SubmitForm, Formspree) или EmailJS. Никакого PHP на статическом сайте | Сразу при добавлении PHP — меняется архитектура |

## Security Mistakes

Domain-specific security issues beyond general web security.

| Mistake | Risk | Prevention |
|---------|------|------------|
| `.htaccess` допускает листинг директорий | Злоумышленник видит ВСЕ файлы на сервере при заходе на `/images/` | `Options -Indexes` в `.htaccess` |
| Файлы бэкапов на сервере: `index-backup.html`, `style-old.css` | Доступны по прямым ссылкам, индексируются поисковиками, раскрывают старые версии | Не хранить бэкапы в `public_html`. Хранить локально или в папке выше уровнем |
| `robots.txt` отсутствует | Поисковики индексируют служебные файлы, битые ссылки, черновики | Создать `robots.txt` даже для заглушки: `User-agent: *` `Allow: /` |
| Пароль FTP сохранён в FTP-клиенте без мастер-пароля | Уходя из-за компьютера, любой может залить файлы на твой сайт | Использовать SFTP вместо FTP, настроить мастер-пароль в клиенте |
| Панель управления Beget без двухфакторной аутентификации | При компрометации пароля — полный доступ к хостингу, доменам, почте | Включить 2FA в настройках аккаунта Beget |
| Версионные файлы на сервере (`style.css?ver=1.2.3`) | Не влияет на безопасность напрямую, но это технический долг | Версионирование в имени файла или через `.htaccess` кэш-бастинг |

## UX Pitfalls

Common user experience mistakes in this domain.

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Текст «Сайт в разработке» вместо контента | Посетитель уходит и не возвращается | Даже заглушка должна содержать: кто ты, чем занимаешься, как связаться. «В разработке» = пустая трата трафика |
| Нет favicon | Вкладка браузера выглядит как «мёртвая», сайт кажется недоделанным | Добавить `favicon.ico` (32×32) в корень сайта. Даже для заглушки |
| Ссылка на email вида `mailto:email@example.com` без защиты от спама | Спам-боты собирают email за минуты после индексации | Обфускация: `email[at]example.com` или JavaScript-декодер |
| Некликабельный телефон на мобильной версии | Пользователь не может позвонить в одно нажатие | `<a href="tel:+79991234567">+7 (999) 123-45-67</a>` |
| Отсутствие Open Graph тегов | При шаринге в соцсетях — пустая ссылка без картинки и описания | Добавить `<meta property="og:title">`, `og:description`, `og:image` |
| Разный дизайн на www и без www | Один и тот же сайт открывается как два разных — путаница | В `.htaccess` выбрать канонический домен: `RewriteCond %{HTTP_HOST} ^www\.(.+)$ [NC]` `RewriteRule ^(.*)$ https://%1/$1 [R=301,L]` |
| Ссылка «написать в Telegram» без `tg://` протокола | Открывается web.telegram.org вместо приложения | `tg://resolve?domain=@username` для мобильных, web-ссылка для десктопа |

## "Looks Done But Isn't" Checklist

Things that appear complete but are missing critical pieces.

- [ ] **Favicon:** Проверить, что отображается во вкладке браузера, а не стандартная иконка Beget
- [ ] **Кодировка:** Открыть сайт на телефоне и десктопе — русский текст везде читаемый
- [ ] **HTTPS:** `https://домен.ru` открывается без предупреждений безопасности
- [ ] **www → без www:** `www.домен.ru` редиректит на `домен.ru` (или наоборот), а не показывает дубликат
- [ ] **404 страница:** При заходе на `домен.ru/несуществующая-страница` показывается твоя страница, а не дефолтная ошибка Apache
- [ ] **robots.txt:** `домен.ru/robots.txt` существует и отдаёт корректные директивы
- [ ] **Сжатие изображений:** Все картинки весят < 200 КБ. Проверить в DevTools → Network → Img
- [ ] **Мобильная вёрстка:** На телефоне (реальном!) нет горизонтального скролла, текст читаемый, кнопки нажимаются
- [ ] **Open Graph:** При вставке ссылки в Telegram/VK/WhatsApp — есть картинка и описание
- [ ] **Файлы на сервере:** В `public_html` только нужные файлы, нет `test.html`, `old-index.html`, `backup/`
- [ ] **FTP-доступ:** Проверен пароль, клиент настроен, загрузка работает. Пароль нигде не светится в репозитории

## Recovery Strategies

When pitfalls occur despite prevention, how to recover.

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Файлы не в `public_html` | LOW | Переместить файлы в правильную папку через файловый менеджер Beget или FTP. Время: 2 минуты |
| Кракозябры (кодировка) | LOW | Заново сохранить HTML в UTF-8 без BOM, добавить `.htaccess` с `AddDefaultCharset UTF-8`. Время: 5 минут |
| Старая версия в кэше | LOW | Очистить кэш браузера (Ctrl+Shift+Del), проверить в приватном режиме. Если не помогло — проверить `.htaccess` на директивы кэширования. Время: 5-15 минут |
| Mixed Content после SSL | MEDIUM | Найти все `http://` в коде через поиск по проекту, заменить на `https://` или `//`. Проверить внешние ресурсы. Время: 15-30 минут |
| Сайт не открывается после смены DNS | LOW | Проверить `ping домен`. Если пингуется старый IP — ждать (до 24ч). Если не пингуется вообще — проверить NS-записи у регистратора. Время: 5 минут + ожидание |
| Мусорные файлы на хостинге | LOW | Сравнить локальную папку и `public_html`, удалить лишнее через файловый менеджер Beget. Время: 10 минут |
| FTP не подключается | LOW | Проверить: хост (обычно `ftp.домен.ru` или IP), порт (21 для FTP, 22 для SFTP), логин/пароль. Если IP заблокирован — разблокировать в панели Beget. Время: 5-15 минут |
| Тяжёлые изображения | MEDIUM | Прогнать все изображения через Squoosh или ImageMagick, перезалить. Если уже много страниц — выше затраты. Время: 20-60 минут |

How roadmap phases should address these pitfalls.

| Pitfall | Prevention Phase | Verification |

| Кодировка (кракозябры) | Phase 1 (Landing Page) | Открыть сайт на десктопе и телефоне — русский текст читаемый во всех браузерах |

| Отсутствие `.htaccess` | Phase 1 (Landing Page) | `https://` работает, 404-страница кастомная, листинг директорий запрещён |
| Тяжёлые изображения | Phase 1 (Landing Page) | Lighthouse Performance > 90, размер страницы < 1 МБ |
| Файловый мусор на сервере | Phase 1 (Landing Page) | В файловом менеджере Beget — только актуальные файлы из проекта |
